# task4 — 项目详情页补全 + 澄清问题入口最小闭环

## 目标
在已完成 task3 的基础上，把“项目创建后进入工作区”再往前推进一步：

- 让 `/workspace/[id]` 不只是展示项目信息
- 增加一个最小的“澄清问题入口区”
- 先不接真实 AI，但要把后续 AI 澄清流程的页面骨架和数据结构入口搭好
- 让用户从“写下想法”正式进入“准备澄清”的状态

---

## 本次范围

### 需要完成
1. **Project 模型补充最少字段**
   - 支持区分项目当前阶段，例如：
     - `draft`
     - `clarifying`
2. **项目详情读取接口（可选但建议做）**
   - `GET /api/projects/[id]`
3. **项目状态更新接口**
   - `PATCH /api/projects/[id]`
   - 本次只需要支持把项目从 `draft` 推进到 `clarifying`
4. **工作区页面增强**
   - `/workspace/[id]` 增加一个“下一步”区域
   - 提供一个按钮：
     - `开始澄清需求`
   - 点击后把项目状态更新为 `clarifying`
5. **澄清页骨架**
   - 新增页面：`/workspace/[id]/clarify`
   - 页面先不接 AI
   - 但要展示：
     - 当前项目标题
     - 当前项目想法
     - 一组固定写死的澄清问题（先静态）
6. **固定问题列表**
   - 先不存数据库
   - 先在代码里维护一个最小静态数组
   - 建议 4~6 个问题，例如：
     - 这个产品主要给谁用？
     - 你最想先解决什么问题？
     - 用户使用它时第一步会做什么？
     - 你希望这个产品最简单的可用版本包含哪些功能？
     - 你更想先做网页、表单工具，还是内容型产品？
7. **状态与异常处理**
   - 项目不存在时继续保持友好提示
   - 状态更新失败时给出提示
   - clarify 页面在项目不存在时也要有友好提示

### 不要做
- 不要接入真实 AI 问答
- 不要保存用户对澄清问题的回答
- 不要生成设计书
- 不要做任务拆解系统
- 不要做项目列表页
- 不要引入登录鉴权
- 不要做大范围重构

## 推荐实现方式

### 一、数据层
如果当前 `Project.status` 已经存在且是字符串，可直接沿用；本 task 只要求把状态流转真正用起来。

推荐最小状态流：
- 创建项目后：`draft`
- 点击“开始澄清需求”后：`clarifying`

如果你认为需要补一个极少量字段，也只能加最小必要字段，不要扩展。

## API 约定

### 1. `GET /api/projects/[id]`（建议做）
成功返回：

```json
{
  "success": true,
  "project": {
    "id": "...",
    "title": "...",
    "idea": "...",
    "status": "draft",
    "createdAt": "..."
  }
}
```

失败返回：

```json
{
  "success": false,
  "error": "项目不存在。"
}
```

### 2. `PATCH /api/projects/[id]`
请求体：

```json
{
  "status": "clarifying"
}
```

成功返回：

```json
{
  "success": true,
  "project": {
    "id": "...",
    "status": "clarifying"
  }
}
```

失败返回：

```json
{
  "success": false,
  "error": "无效状态。"
}
```

### PATCH 规则
- 本次只允许把状态更新为 `clarifying`
- 不要开放任意字段更新
- 参数校验保持简单明确

## 页面要求

### 1. `/workspace/[id]`
在 task3 的基础上新增：

- “下一步”卡片/面板
- 一段简短说明
- `开始澄清需求` 按钮

点击逻辑：
1. 调用 `PATCH /api/projects/[id]`
2. 更新成功后跳转到：
   - `/workspace/[id]/clarify`

页面仍要保留：
- 项目名称
- 项目想法
- 状态
- 创建时间

如果项目已经是 `clarifying`，按钮文案可以变成：
- `继续澄清需求`
或直接跳 clarify 页面。

### 2. `/workspace/[id]/clarify`
这是本 task 的核心新增页面。

页面至少包含：
- 页面标题，例如：`先把你的需求说清楚`
- 一段说明文案
- 项目标题
- 项目想法
- 固定澄清问题列表
- 一个占位说明，告诉用户后续这里会接 AI 引导式澄清

展示方式尽量简单：
- 每个问题做成卡片、编号列表或 panel 均可
- 不需要回答输入框也可以
- 如果你觉得加一个只读占位输入区更利于后续扩展，也可以，但不要复杂化

建议文案示例：
- 标题：`先把需求澄清，再开始做`
- 说明：`下一步我们会像项目教练一样，一步步帮你把模糊想法变成可执行方案。现在先从这些关键问题开始。`
