# task6 — 设计书预览页（静态生成版）

## 目标
在 task5 已完成的基础上，新增一个**设计书预览闭环**：

- 用户已经创建项目
- 用户已经完成需求澄清并保存回答
- 系统基于已有的 `title`、`idea`、`clarification.answers`，生成一份**静态拼装**的设计书预览内容
- 用户可以在 Web 页面中查看这份设计书预览

这次**不要接真实 AI**，不要做真正的大模型生成，只做“基于已有数据的结构化整理 + 页面展示”。

## 设计原则

本次设计书不是“智能生成”，而是“静态整理”。

也就是说，你要把已有数据按固定规则拼成一份**看起来像设计书**的结构化页面，先把产品闭环打通。

目标是让零基础用户感受到：
- 我先写想法
- 再回答澄清问题
- 然后系统能给我一份更清楚的项目设计书预览

---

## 数据来源

设计书预览只允许使用这些已有数据：

- `project.title`
- `project.idea`
- `project.status`
- `project.clarification.answers`
- 已有固定澄清问题配置

不要引入新的复杂数据表，除非你认为必须新增一个极小的类型文件或纯函数文件来整理输出。

## 建议实现方式

### 1. 新增设计书整理函数
建议新增类似文件：
- `lib/server/design-brief.ts`

职责：
- 输入：Project + clarification answers
- 输出：结构化 design brief 对象

例如输出结构可类似：

```ts
{
  summary: string,
  targetUsers: string,
  coreProblem: string,
  coreFeatures: string[],
  mvpScope: string[],
  outOfScope: string[],
  nextSteps: string[]
}
```

可以根据 task5 中的固定问题 key 做映射整理。

### 2. 新增页面
- `app/workspace/[id]/design/page.tsx`

页面职责：
- 读取项目
- 判断项目是否存在
- 判断项目是否已完成 clarified
- 若满足条件，展示设计书预览

### 3. workspace 入口
在 `app/workspace/[id]/page.tsx` 中：
- 如果项目状态为 `clarified`
- 显示“查看设计书预览”按钮或链接

## 页面要求

### `/workspace/[id]/design`
建议包含这些区块：

1. 页面标题
   - 例如：`项目设计书预览`
2. 项目基本信息
   - 项目名称
   - 当前状态
3. 一句话概述
4. 目标用户
5. 核心问题
6. 核心功能
7. MVP 范围
8. 非本期范围
9. 下一步建议
10. 返回工作区入口

### 内容生成规则
内容允许“半模板化”，但不要写得太死板。

例如：
- 一句话概述：可由 `title + idea` 拼接压缩得到
- 目标用户：优先取澄清回答中的目标用户相关内容
- 核心功能：从回答中拆成 2-5 条列表
- MVP 范围：用固定模板 + 回答内容整理
- 非本期范围：固定给出 2-4 条常见非 MVP 范围提示
- 下一步建议：固定输出 3 条左右即可

注意：
- 不要求 NLP 很强
- 只要结构清楚、页面可用、逻辑成立即可
