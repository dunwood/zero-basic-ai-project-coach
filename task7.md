# task7 — 从设计书预览生成“可执行任务清单”静态版

## 目标
在 task6 已完成的基础上，把设计书预览进一步整理成一个 **静态任务清单页**，让零基础用户看到“接下来该做什么”。

本次仍然**不接真实 AI**，只基于当前已有的：
- Project 基本信息
- 已保存的澄清回答
- task6 的设计书整理逻辑

生成一份结构化的任务清单页面。

---

## 本次范围

### 需要完成
1. 新增 `/workspace/[id]/tasks` 页面
2. 基于现有项目数据生成**静态任务清单结构**
3. 在 `/workspace/[id]` 增加“查看任务清单”入口
4. 在 `/workspace/[id]/design` 增加“进入任务清单”入口
5. 处理三种状态：
   - `clarified`：可正常查看任务清单
   - 未完成澄清：提示先完成澄清
   - 项目不存在：友好提示
6. 任务清单内容先做静态规则整理，不做保存，不做勾选，不做数据库新增表

### 不要做
- 不要接真实 AI
- 不要调用外部 API
- 不要新增 Task 数据表
- 不要做任务编辑/拖拽/完成状态
- 不要做任务清单持久化
- 不要做设计书导出
- 不要做项目列表页
- 不要做登录系统
- 不要做无关重构

---

## 页面目标

### `/workspace/[id]/tasks`
目标：让用户看到一份“下一步可执行”的任务拆解清单。

建议页面结构：
- 页面标题：`项目任务清单`
- 一段简短说明：这是基于当前设计书整理出的第一版执行任务
- 按阶段展示任务卡片

建议拆成 4 个阶段：
1. **项目准备**
2. **页面与交互实现**
3. **数据与接口实现**
4. **验证与上线准备**

每个阶段至少包含：
- 阶段标题
- 阶段说明
- 2~5 条任务项

每条任务项建议包含：
- 标题
- 简短说明
- 可选的小标签（如：页面 / 接口 / 数据 / 验证）

## 静态任务整理逻辑

你可以新增一个类似 `lib/server/task-plan.ts` 的整理文件。

输入：
- project.title
- project.idea
- clarification.answers
- task6 已有 design brief 整理结果

输出：
- 一个静态任务计划对象，例如：
  - `summary`
  - `phases`
  - `phases[].title`
  - `phases[].description`
  - `phases[].items[]`

### 生成原则
不要复杂化，不要引入大算法。只做**规则化整理**。

例如可以固定生成这些阶段：

#### 1. 项目准备
示例任务：
- 明确这个产品要先服务哪一类用户
- 确定首页或首个入口页要展示什么
- 确定最小可用版本只保留哪些功能

#### 2. 页面与交互实现
示例任务：
- 搭建首页/输入页/结果页
- 补齐关键按钮和跳转路径
- 明确空状态、报错状态、加载状态

#### 3. 数据与接口实现
示例任务：
- 确定需要保存哪些核心字段
- 编写创建/读取/更新接口
- 处理表单提交和数据回填

#### 4. 验证与上线准备
示例任务：
- 做本地自测
- 检查关键路径是否跑通
- 准备部署前检查项

你可以根据 design brief 的内容，把任务标题或说明做轻微定制，但不要扩展成真正智能规划。

### `/workspace/[id]`

## 状态处理

### 1. clarified
- 可以正常查看 `/workspace/[id]/tasks`
- 页面展示完整任务清单

### 2. 未 clarified
- 显示提示：`你还没有完成需求澄清`
- 提供入口：
  - 返回工作区
  - 去完成需求澄清

### 3. not found
- 显示提示：`没有找到这个项目`
- 提供入口：
  - 回到项目创建页
